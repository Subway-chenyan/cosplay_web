# 首页与比赛详情页服务端筛选设计

## 背景

当前筛选链路存在两类问题：

- 首页新版界面只保留关键词搜索，旧的 `VideoFilters` 未挂载；旧筛选参数还存在前后端数组编码不一致、`styleTag`/`ipTag` 未完整传递等问题。
- 比赛详情页一次请求最多 1000 个视频及全部获奖记录，再由浏览器按年份或奖项筛选。数据量超过上限时会漏数据，初始载荷持续增长，视频、获奖记录和分页状态也容易不一致。

本设计将首页和比赛详情页的业务筛选统一迁移到后端。前端只维护筛选草稿、URL 状态、请求状态和展示分组，不再依靠预加载全量数据保证正确性。

## 目标

### 首页

- 支持关键词、年份、比赛和社团组合筛选。
- 年份单选；比赛、社团支持搜索多选。
- 同一多选维度内部使用 OR，不同维度以及关键词之间使用 AND。
- 筛选条件与分页由后端处理，响应提供准确总数。
- 已应用条件同步到 URL，可刷新、分享，并支持浏览器前进/后退。

### 比赛详情页

- 支持年份、奖项分别筛选或同时筛选。
- 后端统一筛选获奖视频、无视频获奖记录和未获奖视频。
- 使用“加载更多”，失败时保留已加载内容并可重试。
- 筛选条件同步到 URL。

### 可靠性

- 防止旧请求晚返回后覆盖新筛选结果。
- 使用确定性排序，避免连续页面发生不必要的重复和跳动。
- 筛选选项独立于当前结果集，不因筛选为空或分页未加载完而消失。
- 区分“没有结果”和“请求失败”。

## 非目标

- 本次不恢复 IP、标签或风格筛选。
- 本次不改变智能检索的查询和结果协议。
- 本次不重做比赛、社团或视频的数据模型。
- 本次不引入通用查询语言、GraphQL 或独立搜索引擎。
- 本次不修改现有视觉主题，只新增与现有风格一致的筛选控件和状态反馈。

## 总体架构

两页共享明确的筛选参数语义，但使用各自适合展示模型的接口：

1. 首页继续使用视频列表接口。后端在 `Video` 查询集上完成关键词、年份、比赛、社团、排序和分页。
2. 比赛详情页新增专用参赛条目接口。后端把获奖记录和未获奖视频规范化为统一条目，再筛选、排序和分页。
3. 首页视频结果与比赛参赛条目使用独立前端状态，避免当前共享 `videosSlice` 导致跨页面结果互相覆盖。
4. URL 表示“已应用筛选”；控件内部草稿与已应用状态分离。

## 筛选协议

### 通用规则

- 空字符串、空数组和未定义参数不发送。
- UUID 参数必须合法；年份必须是合理整数；页码和页面大小必须为正整数。
- 非法参数返回 HTTP 400 和字段级错误，不静默降级为全量结果。
- 多选参数采用逗号分隔，例如 `competitions=id-a,id-b`。前后端只保留这一种编码方式。
- 服务端对重复 ID 去重。

### 首页视频接口

请求示例：

```http
GET /api/videos/?search=原神&year=2025&competitions=id-a,id-b&groups=id-x,id-y&page=1&page_size=12
```

参数：

| 参数 | 类型 | 语义 |
| --- | --- | --- |
| `search` | string | 关键词 |
| `year` | integer | 年份精确匹配 |
| `competitions` | UUID CSV | 任一比赛匹配 |
| `groups` | UUID CSV | 任一社团匹配 |
| `page` | integer | 页码 |
| `page_size` | integer | 固定为 12，后端仍执行上限校验 |

关键词覆盖视频标题、描述、BV 号、社团名、比赛名、奖项名和获奖记录剧目名。涉及获奖记录的一对多连接后，查询必须 `distinct()`，保证一个视频只出现一次且 `count` 准确。

响应沿用现有分页结构：

```json
{
  "count": 42,
  "next": "...page=2",
  "previous": null,
  "total_pages": 4,
  "current_page": 1,
  "page_size": 12,
  "results": []
}
```

默认排序为 `-created_at, -id`。第二排序键用于处理相同创建时间，保证翻页顺序确定。

### 首页筛选选项

- `GET /api/videos/filter-options/` 返回数据库中存在视频的年份列表和数量。
- 比赛选择器使用 `GET /api/competitions/competitions/?search=<keyword>&page_size=20`，后端为比赛列表启用名称和描述搜索。
- 社团选择器复用 `GET /api/videos/search-groups/?search=<keyword>&page_size=20`。
- 已选比赛和社团必须保留完整标签，即使它们不在当前搜索返回的前 20 项中；缺失项按 ID 补取。

年份响应示例：

```json
{
  "years": [
    { "value": 2026, "count": 120 },
    { "value": 2025, "count": 98 }
  ]
}
```

### 比赛详情筛选选项

新增：

```http
GET /api/competitions/competitions/{competition_id}/filter-options/
```

响应返回该比赛完整年份、完整奖项及未筛选总数。年份取 `CompetitionYear`、视频年份和获奖记录年份的并集，避免历史数据尚未完全规范化时选项缺失。奖项按现有比赛配置排序；后端提供原始列表和数量，前端继续应用已有展示顺序配置。

```json
{
  "years": [{ "value": 2025, "count": 36 }],
  "awards": [{ "id": "...", "name": "最佳舞台表现奖", "count": 8 }],
  "total_count": 54
}
```

选项统计独立于当前筛选，选择一个条件后其他可选项不会消失。

## 比赛参赛条目接口

新增：

```http
GET /api/competitions/competitions/{competition_id}/entries/?year=2025&award=award-id&page=1&page_size=24
```

`year` 和 `award` 均可省略，也可同时存在。奖项必须属于 URL 中的比赛，否则返回 400。

### 统一条目

接口把以下三类数据转为统一条目：

1. `awarded_video`：获奖记录有关联视频。
2. `award_without_video`：获奖记录没有关联视频。
3. `unawarded_video`：比赛视频没有任何获奖记录。

响应条目：

```json
{
  "entry_id": "award-record:<uuid>",
  "kind": "awarded_video",
  "year": 2025,
  "award": { "id": "...", "name": "最佳舞台表现奖" },
  "award_record": { "id": "...", "drama_name": "...", "description": "..." },
  "video": { "id": "...", "title": "..." }
}
```

未获奖视频的 `entry_id` 为 `video:<uuid>`，`award` 与 `award_record` 为 `null`。无视频获奖记录的 `video` 为 `null`。

同一视频获得多个奖项时，每条获奖记录分别产生一个条目，这是现有“按奖项分组”展示所需行为，不视为重复数据。

### 查询与分页实现

- 获奖记录分支先限定比赛，再应用年份和奖项条件。
- 未获奖视频分支限定比赛并排除存在获奖记录的视频；选择奖项时该分支为空。
- 两个分支投影为相同的轻量字段后使用 `UNION ALL`，按年份倒序、创建时间倒序、条目类型和唯一 ID 排序。
- 分页发生在统一条目集合上，而不是分别分页后由前端拼接。
- 分页结果再批量加载所需视频、奖项和获奖记录，避免逐条查询。
- 页面大小默认 24，最大 100。

响应：

```json
{
  "count": 54,
  "next": "...page=2",
  "previous": null,
  "results": []
}
```

前端使用服务端返回的 `next` 判断是否还能加载，不通过“本页少于 page_size”自行推断。

## 前端设计

### 首页筛选区

新增首页筛选组件，包含：

- 关键词输入框：沿用现有普通搜索输入。
- 年份单选。
- 比赛可搜索多选。
- 社团可搜索多选。
- “应用筛选”和“清空筛选”。
- 已应用条件摘要和准确结果数。

比赛、社团选择变更只修改本地草稿。点击“应用筛选”或在关键词框回车后，一次性写入 URL 并发起一次请求，避免用户连续多选时产生多次无效请求。清空操作清空草稿、URL 和页码。

### 比赛详情筛选区

- 年份与奖项不再使用互斥的视图模式。
- 两个条件都允许为空，可任意组合。
- 条件变化立即写入 URL，清空当前条目并请求第一页。
- “清除筛选”一次清除两个参数。
- “加载更多”只请求响应中的 `next`，成功后按 `entry_id` 去重追加。
- 前端对已加载条目做纯展示分组：有奖项条目按奖项分组，无奖项条目进入“参与作品”。前端不再决定条目是否满足筛选。

### 状态边界

建立独立状态：

- `homeVideos`：结果、分页、已应用筛选、首次加载/重新筛选状态和错误。
- `competitionEntries`：按比赛 ID 保存条目、`next`、总数、筛选状态、加载更多状态和错误。
- 筛选选项状态可独立缓存，不与结果列表共用生命周期。

现有 `videosSlice` 可继续服务视频详情、社团详情等页面，但首页和比赛详情不再共享同一个 `videos` 数组。

### URL 规则

首页：

```text
/?q=关键词&year=2025&competitions=id-a,id-b&groups=id-x,id-y&page=2
```

比赛详情：

```text
/competitions/{id}?year=2025&award=award-id
```

- URL 是已应用筛选的唯一持久来源。
- 首次进入、刷新和 `popstate` 时从 URL 解析状态并请求。
- 写入 URL 时使用稳定参数顺序与去重后的 ID，减少无意义历史记录。
- 重新筛选时页码重置为 1。

### 请求竞态

- 服务层接受 `AbortSignal` 并传给 Axios。
- 新筛选开始时取消该页面尚未完成的旧请求。
- reducer 记录当前请求 ID；即使取消未及时生效，也只接受当前请求的响应。
- 页面卸载时取消页面级请求。

## 加载与错误状态

### 首页

- 首次加载没有内容时显示骨架屏。
- 重新筛选时保留旧结果并加淡化加载遮罩；新响应成功后原子替换。
- 重新筛选失败时保留旧结果，显示错误和“重试”按钮，不显示“0 条结果”。
- 只有成功响应且 `count=0` 时显示空结果。

### 比赛详情

- 首次加载失败显示页面级重试。
- 加载更多失败保留所有已加载分组，按钮变为“重试加载”。
- 重试使用同一个 `next` URL，成功前不推进页码。
- 筛选条件无效时展示字段级提示，并允许一键清除失效条件。

## 性能与数据库

- 视频模型现有 `year`、`group`、`competition` 及组合索引可覆盖主要筛选。
- 获奖记录已有 `competition_year + award`、`video + competition_year` 索引。
- 实现时使用 `select_related`/`prefetch_related` 和批量 hydration，避免 N+1。
- 通过查询数量测试约束首页与比赛条目接口的数据库查询次数。
- 若实际执行计划显示比赛归属连接成本明显，再根据 `EXPLAIN ANALYZE` 增补索引；设计阶段不预先添加未经验证的索引。

## 测试策略

### 后端

- 首页每个单条件筛选。
- 关键词与年份、比赛、社团组合。
- 比赛/社团多选内部 OR，不同维度 AND。
- 关联奖项关键词不会制造重复视频或错误计数。
- 非法年份、UUID、页码和不属于当前比赛的奖项返回 400。
- 比赛条目三种类型都能返回。
- 比赛条目按年份、奖项及组合筛选正确。
- 同一视频多奖项按不同获奖记录返回。
- 统一分页无重复，排序在固定数据集上稳定。
- 筛选选项不会受当前结果分页影响。

### 前端

- URL 到筛选草稿和请求参数的解析。
- 应用筛选只发起一次请求，清空时重置页码。
- 浏览器前进/后退恢复筛选和结果。
- 旧响应不能覆盖新筛选。
- 加载更多追加并按 `entry_id` 去重。
- 加载更多失败保留结果并能重试。
- 成功空结果与请求失败使用不同状态。

### 浏览器验证

- 首页分别验证关键词、年份、比赛多选、社团多选和组合筛选。
- 刷新以及前进/后退后条件与结果保持一致。
- 比赛详情验证年份、奖项、组合、清空和连续加载更多。
- 检查移动端筛选面板、键盘操作、加载状态、错误状态和控制台错误。

## 兼容与迁移

- 保留现有视频列表基础路径与分页响应，减少其他调用方受影响。
- `VideoFilter` 暂时可保留旧标签参数，但首页新组件不会发送；待确认无调用方后另行删除，不纳入本次范围。
- 比赛详情切换到新 `entries` 接口后，移除该页面一次拉取 1000 条视频/获奖记录及本地筛选逻辑。
- 不在同一改动中删除仍被视频详情、社团详情使用的旧 thunk。

## 验收标准

1. 首页关键词、年份、比赛多选和社团多选可以任意组合，结果与总数来自后端且准确。
2. 首页筛选不依赖客户端已有视频集合，翻页后条件保持不变。
3. 比赛详情的年份与奖项可单独或组合使用，三类参赛条目都不会因 1000 条上限而遗漏。
4. 比赛详情“加载更多”不会清空已有结果；失败可重试，成功后无重复条目。
5. 两页刷新、分享链接、浏览器前进/后退均能恢复筛选。
6. 快速连续改变筛选不会出现旧结果覆盖新结果。
7. 后端与前端自动化测试通过，浏览器端无错误覆盖层和新增控制台错误。
